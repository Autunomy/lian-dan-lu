# 电商计费与结算的边界

## 1. 目标与核心问题

在电商场景中，`计费` 和 `结算` 经常被混用，导致系统职责不清、资金风险上升、对账困难。  
本文聚焦三个问题：

- 计费与结算的领域边界如何划分
- 资金流如何编排（清分、冻结、出款）
- 出入款方信息如何建模与治理

---

## 2. 一句话定义

- `计费（Billing）`：计算应收应付，产出“账务事实”
- `结算（Settlement）`：执行清分与出入款，产出“资金事实”

可理解为：

- 计费回答：谁应该给谁多少钱，依据是什么
- 结算回答：钱何时、通过什么通道、从哪个账户到哪个账户

---

## 3. 领域边界（DDD 口径）

## 3.1 计费域（Billing Context）

### 职责

- 订单金额构成计算：商品价、运费、优惠、税费、服务费、佣金、分润基数
- 计费事件处理：下单、发货、签收、退款、拒付、补差、调账
- 费率规则引擎：按商户、品类、合同版本、生效时间计算
- 应收应付生成：平台应收、商家应收、渠道手续费、代理分成等
- 账务凭证沉淀：形成不可篡改的计费明细

### 输入

- 业务事件（订单、售后、物流）
- 合同与费率配置
- 活动与优惠规则

### 输出

- `BillingStatement`（计费单）
- `BillingLine`（计费明细行）
- `AR/AP`（应收/应付余额变化）

### 不做什么

- 不做银行打款
- 不直接处理支付通道回单
- 不做对外资金指令下发

## 3.2 结算域（Settlement Context）

### 职责

- 结算周期处理：T+0 / T+1 / T+N
- 清分与轧差：按规则把“应收应付”转换为“待结算净额”
- 出入款执行：发起付款、接收回执、失败重试
- 渠道对账：内部账 vs 渠道账 vs 银行账
- 风险控制：冻结、延迟结算、限额、黑白名单

### 输入

- 计费单与应收应付明细
- 通道能力与账户状态
- 结算策略（周期、最小出款额、保留金规则）

### 输出

- `SettlementOrder`（结算单）
- `PayoutOrder`（出款单）
- `ReconciliationReport`（对账报告）

### 不做什么

- 不修改计费规则
- 不反向重算订单金额

---

## 4. 资金流编排（参考流程）

以平台型电商为例：

1. 买家支付成功，资金进入平台收单账户（或监管备付金账户）
2. 计费域生成应收应付（商家货款、平台服务费、渠道费、分润）
3. 结算域按策略清分：
   - 扣除渠道手续费
   - 扣除平台服务费
   - 计算商家净应付
   - 处理分润与代扣
4. 到账条件满足后触发出款（通常 T+1 或更长）
5. 银行/通道返回回单，更新出款状态
6. 做日终对账，生成差异清单并触发补偿

---

## 5. 出入款方信息模型（关键）

建议把“资金参与方”抽象为统一模型：`FundParty` + `FundAccount`。

## 5.1 主体信息 FundParty

- `party_id`：主体唯一 ID
- `party_type`：PLATFORM / MERCHANT / SUB_MERCHANT / PARTNER / CHANNEL
- `party_name`：主体名称
- `id_type`：统一社会信用代码/身份证/其他
- `id_no`：证件号
- `kyc_status`：KYC 状态
- `risk_level`：风险等级
- `status`：ACTIVE / SUSPENDED / CLOSED

## 5.2 账户信息 FundAccount

- `account_id`：账户唯一 ID
- `party_id`：归属主体
- `account_type`：RECEIVE / PAY / BOTH
- `channel`：银行/支付渠道
- `channel_account_no`：通道账号或商户号
- `currency`：币种
- `country`：国家/地区
- `can_inbound`：是否可入款
- `can_outbound`：是否可出款
- `daily_limit`：日限额
- `single_limit`：单笔限额

## 5.3 出入款关系（指令层）

- `payer_party_id` / `payer_account_id`
- `payee_party_id` / `payee_account_id`
- `amount` / `currency`
- `biz_order_no`（业务幂等键）
- `instruction_no`（资金指令号）
- `purpose_code`（用途：货款/佣金/退款等）

## 5.4 过渡户（Transit Account）标识与边界

场景：A 待结算户 -> B 现金户 -> C/D/E 现金户。  
其中 B 不是最终收益方，而是中转方（过渡户）。

### 结论

- 需要有“过渡户标识”告诉结算系统存在中转意图
- 但不应让计费下沉到“执行编排细节”（如通道优先级、重试、切路）

换句话说：

- 计费负责表达 `资金分配意图`
- 结算负责落地 `资金执行路径`

### 推荐做法：计费输出分账腿（Allocation Legs）

计费输出里直接携带资金腿信息，而不是只给净额：

- leg1: A_SETTLE -> A_CASH, amount = x
- leg2: A_SETTLE -> B_CASH, amount = y
- leg3: B_CASH -> C_CASH, amount = y * p1
- leg4: B_CASH -> D_CASH, amount = y * p2
- leg5: B_CASH -> E_CASH, amount = y * p3

并在 B 相关腿上加业务语义字段：

- `role = TRANSIT`（过渡户角色）
- `pass_through = true`（过路资金）
- `retain_amount = 0`（如需全额分出）
- `next_allocation_group = group_id`（后续分配组）

### 为什么合理

- 不加标识：结算系统可能把 B 当最终收款方，造成“只到 B 不继续分”
- 标识过细：如果计费指定“具体通道、具体重试策略”，会侵入结算执行域

因此最佳边界是：

- 计费给“可执行的意图与约束”
- 结算决定“执行策略与异常处理”

### 建议字段（计费 -> 结算）

- `allocation_group_id`：同一组分配链路 ID
- `from_account_id` / `to_account_id`
- `amount` / `currency`
- `node_role`：NORMAL / TRANSIT / FINAL_BENEFICIARY
- `must_pass_through`：是否必须经过该节点
- `ratio`：比例（可选）
- `rounding_rule`：尾差规则（如 LAST_BENEFICIARY）
- `dependency_leg_id`：依赖前置腿（保证先到 B 再分 C/D/E）

### 结算侧必须校验

- 金额守恒：进入 B 的金额 == B 分出的金额（全额过路时）
- 拓扑合法：存在环路/断链要拒绝执行
- 账户可用：B/C/D/E 账户状态、限额、KYC、冻结状态
- 顺序约束：未完成前置腿不得执行后续腿

### 边界红线（再次强调）

- 计费可以感知“是否过渡户、是否全额分出”
- 计费不应感知“如何调用银行/支付通道、失败重试几次、路由切换策略”

## 6. 核心数据对象与表结构建议

## 6.1 计费侧

- `billing_statement`
  - statement_id, biz_type, biz_id, status, rule_version, occurred_at
- `billing_line`
  - line_id, statement_id, subject(平台/商家/渠道), fee_type, amount, direction(DR/CR)
- `receivable_payable_ledger`
  - ledger_id, subject_id, balance, frozen_balance, updated_at

## 6.2 结算侧

- `settlement_batch`
  - batch_id, period, status, created_at
- `settlement_order`
  - settlement_id, batch_id, party_id, gross_amount, fee_amount, net_amount, status
- `payout_order`
  - payout_id, settlement_id, payer_account_id, payee_account_id, amount, status, channel_txn_no
- `recon_diff`
  - diff_id, source_type, source_no, expected_amount, actual_amount, diff_reason, status

---

## 7. 接口边界（防越界）

计费 -> 结算只通过“确定性账务结果”交互，不传规则细节：

- `POST /settlement/orders/create-from-billing`
- `POST /settlement/orders/{id}/confirm`
- `POST /settlement/payouts/{id}/execute`
- `POST /settlement/reconciliation/run`

补充：当存在过渡户时，接口入参应支持 `allocation_legs` 与 `dependency_leg_id`，由结算系统按依赖关系编排执行。

结算 -> 计费只回传结果事件，不回写计费规则：

- `SETTLEMENT_CONFIRMED`
- `PAYOUT_SUCCEEDED`
- `PAYOUT_FAILED`
- `RECON_DIFF_FOUND`

---

## 8. 设计红线（必须遵守）

- 计费域不直接打款
- 结算域不改价、不改费率
- 所有资金动作必须可追溯到计费凭证
- 所有出款必须具备双幂等：业务单号 + 指令号
- 对账必须覆盖三账一致：业务账、资金账、渠道账

---

## 9. 常见坑与治理建议

- `坑1`：先打款后记账  
  - 建议：先生成结算与付款指令，再执行出款，最后回写状态

- `坑2`：退款与结算跨期处理混乱  
  - 建议：按会计期间留痕，使用补扣/补付单，不直接覆盖历史单

- `坑3`：账户信息散落多个系统  
  - 建议：统一账户主数据服务，所有出入款只引用账户 ID

- `坑4`：失败重试造成重复出款  
  - 建议：通道层幂等键 + 内部指令状态机 + 对账反查

---

## 10. 结论

`计费` 负责“算清楚”，`结算` 负责“付清楚”。  
边界清晰后，能显著提升：

- 资金安全性
- 对账效率
- 合规审计能力
- 业务变更速度（规则变更不影响资金执行链路）
